# Coding Standards & Conventions

## 📁 File & Folder Structure
Every new module must follow this exact structure inside `src/modules/[module_name]/`:
- `[module].routes.ts`: Defines Express routes, attaches middleware (auth, validation).
- `[module].controller.ts`: Handles HTTP req/res. Calls services. Returns standardized JSON.
- `[module].service.ts`: Contains business logic. Calls repositories. Throws custom errors.
- `[module].repository.ts`: Contains Prisma database queries. No business logic here.
- `[module].validation.ts`: Zod schemas for request body, params, and query.

## 💻 TypeScript & Code Style
- **Strict Typing:** No `any` types. Define interfaces for all DTOs and Prisma payloads.
- **Async/Await:** Always use async/await. Never use raw `.then()/.catch()`.
- **Error Handling:** Use the global error handling middleware. Throw standardized errors (e.g., `NotFoundError`, `ValidationError`) from services.
- **Naming Conventions:** 
  - Files: `kebab-case.ts` (e.g., `room-type.service.ts`)
  - Variables/Functions: `camelCase`
  - Classes/Interfaces/Types: `PascalCase`
  - DB Models: `PascalCase` (Prisma default)

## 🗄️ Database & Prisma Rules
- **Soft Deletes:** NEVER use `prisma.model.delete()`. Always use `prisma.model.update({ data: { isActive: false } })`.
- **Tenant Isolation:** Every repository method MUST accept and filter by `tenantId` (except for Super Admin operations).
- **Select/Include:** Be explicit with Prisma `select` or `include` to avoid over-fetching data.
- **Transactions:** Use `prisma.$transaction()` when creating/updating multiple related records (e.g., creating a reservation and assigning rooms).

## 🔐 Middleware Usage
- Protect routes with `authenticate` (verifies JWT).
- Protect routes with `requireRole(['RoleName'])` or `requirePermission('permission.name')`.
- Protect routes with `requireTenantAccess` to ensure the user belongs to the requested tenant.
- Validate payloads with `validate(schema)` middleware before hitting the controller.
